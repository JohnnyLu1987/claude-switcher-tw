# fcc-server 修復記錄（2026-06-05）

切換器本身沒問題，問題全在 fcc-server 無法被程式自動啟動 + 連不上免費模型。
根因是**四個疊在一起**的問題，逐一修好。

## 問題與修復

### 1. Python 3.14.0 的 OpenSSL 缺陷（applink 崩潰）
- 症狀：fcc-server 非互動啟動時 `OPENSSL_Uplink: no OPENSSL_Applink` 立刻崩。
- 根因：uv 下載的 `cpython-3.14.0` 那套 OpenSSL build 有 bug，載憑證就崩。
- 修復：用 Python **3.14.5**（OpenSSL 3.5.6）重裝 fcc（套件硬性要求 >=3.14，不能降 3.12）。
  ```
  uv tool install "git+https://github.com/Alishahryar1/free-claude-code.git@c1af215f" --python 3.14.5 --force --system-certs
  ```

### 2. 壞掉的 SSL_CERT_FILE（User 環境變數）
- 症狀：指向已刪除的 `...\Python\Python312\site-packages\certifi\cacert.pem`，害 uv 連不上 pypi。
- 修復：移除該 User 層級環境變數（原值記錄在 backup\removed-SSL_CERT_FILE-原值.txt）。

### 3. tiktoken 啟動時要下載編碼檔，被 TLS 攔截擋住
- 修復：把快取檔複製到永久位置 `C:\Users\JohnnyLu\.fcc\tiktoken-cache`，
  切換器啟動 fcc-server 時設 `TIKTOKEN_CACHE_DIR` 指向它。

### 4. TLS 攔截（防毒/防火牆掃 HTTPS）害 fcc 連不上 NVIDIA
- 症狀：`CERTIFICATE_VERIFY_FAILED`；併入 Windows 憑證後又因 OpenSSL 3.5 嚴格檢查
  （BasicConstraints 未標 critical）而拒絕攔截者的非標準憑證。
- 修復：在 fcc venv 裝 **truststore**，並加 `.pth` 自動注入，讓 fcc 改用 **Windows 憑證驗證**
  （Windows 接受攔截憑證，所以瀏覽器正常；truststore 讓 fcc 一致）。
  - 檔案：`<venv>\Lib\site-packages\zz_truststore_inject.pth`
    內容：`import truststore; truststore.inject_into_ssl()`

## 若重裝 fcc 會遺失的修復（需重做）
- truststore 的安裝與 `.pth`（步驟 4）→ 重裝後要重做。
- 其餘（環境變數、tiktoken 快取、切換器設定）不受重裝影響。

## 驗證結果
- fcc-server 可隱藏、無視窗、程式化自動啟動 ✅
- proxy → NVIDIA nemotron 免費模型 → 正常回應 ✅
